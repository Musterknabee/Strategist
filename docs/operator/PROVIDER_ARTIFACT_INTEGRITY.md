# Provider artifact integrity (paper research)

This runbook defines integrity expectations for provider/sample/research/paper artifacts generated by the provider paper loop.

This runbook complements (does not replace) the core single-tenant sequence in `docs/operator/OPERATOR_EASE_OF_USE_COMMANDS.md` and deployment readiness in `docs/deployment/SINGLE_TENANT_DEPLOYMENT_READINESS.md`.

## Artifact descriptor expectations

Provider/sample/research artifacts should expose descriptor metadata (or `UNKNOWN`/`PENDING` when unavailable):

- `schema_version`, `artifact_id`, `artifact_kind`, `generated_at_utc`
- `command`, `command_args_redacted`
- `code_fingerprint` and/or `git_commit`
- `config_fingerprint`
- provider attribution (`provider_id`, `provider_name`, `provider_mode`)
- trust/license/freshness caveat fields
- digest pointers (`artifact_sha256` / manifest digest fields)
- replay linkage (`replay_manifest_path`, `replayable_offline`)
- `paper_only=true`, `live_trading_blocked=true`
- `warnings[]`, `blockers[]`

## Provider trust/license/freshness caveats

- Public and freemium sources can be useful for research smoke runs but require explicit caveats.
- Missing provider keys must remain action-required and non-fatal (`PENDING_KEY`, `ACTION_REQUIRED`, `UNKNOWN`).
- Artifact presence alone must not imply promotability, production readiness, or live readiness.

Canonical provider/evidence status vocabulary used by operator read-plane payloads:

- `OK`: evidence present and verification checks passed.
- `OPTIONAL_NOT_CONFIGURED`: optional key is not configured; local diagnostics may continue.
- `PENDING_KEY`: provider flow requested but key/broker setup is still pending.
- `UNAVAILABLE`: provider was explicitly unreachable/temporarily unavailable.
- `STALE`: sample/evidence timestamp is older than freshness policy.
- `DEGRADED`: evidence exists but verification or provider quality checks failed.
- `UNKNOWN`: evidence is missing/not yet produced; never interpreted as pass.
- `BLOCKED`: explicit command intent cannot complete without required setup/evidence.

## Read-plane and cockpit expectations

Read-plane payloads should surface replay/integrity summaries:

- replay status (`OK` / `DEGRADED` / `UNKNOWN`)
- input/output artifact counts
- missing artifact count
- digest mismatch count
- paper-only/live-block posture

When replay metadata is absent, payloads should degrade honestly (`UNKNOWN`/`PENDING`), never pass silently. Digest mismatches must surface as degraded/blocked evidence.

Replay verification remains an offline integrity check:

- no provider keys required,
- no network calls,
- no ledger mutation,
- no artifact creation/deletion/mutation,
- no deployment approval or operator signoff authority,
- no live-trading authorization,
- no profitability claims.

## Safety boundary

- Paper-only research evidence is not live execution authority.
- No profitability claims from artifact generation.
- No production-approval or operator-signoff claims from descriptor/replay status alone.
